## 1. 项目概述

### 1.1 项目背景

当前制造企业面临多品种、小批量、短交期的生产环境，传统的人工排产方式已无法满足快速响应和资源优化的需求。生产排程涉及复杂的约束条件下（设备产能、工序顺序、物料齐套、人员技能等）进行多目标优化（准时交付、设备利用率、换线成本、最小库存等），人工排程效率低且难以兼顾全局最优。

本项目旨在搭建一套生产排程智能体，利用运筹优化算法与人工智能技术，以低代码/开源工具为核心，在1-2周内快速构建排程原型系统，验证优化效果后逐步迭代完善。

### 1.2 项目目标

**近期目标（1-2周原型）**
- 搭建一个可运行的排程原型，涵盖工单导入、约束建模、排程计算、甘特图可视化四大核心能力
- 以10-20个工单、3-5台设备、3-5道工序的小规模数据集验证排程效果
- 排程质量不低于有经验的计划员人工排程水平，排程时间<5分钟（vs人工4-6小时）

**远期愿景（迭代完善）**
- 逐步纳入换线成本、模具约束、人员排班、物料齐套等更多约束条件
- 与ERP/MES/WMS系统集成，实现数据自动同步与动态排程
- 通过持续学习优化排程策略，逐步提升设备利用率、交期达成率等关键KPI

### 1.3 适用范围

本PRD适用于生产排程智能体的原型搭建阶段（Phase 1），后续正式产品化阶段需根据原型验证结果单独编写PRD。

## 2. 业务需求

### 2.1 排程业务全景

生产排程覆盖从工单接收、任务分解、资源分配到计划下发的完整流程，核心包括排程目标定义、约束条件管理、基础数据准备和结果输出四大维度。

### 2.2 排程目标（优化目标）

排程优化支持多维目标组合，用户可根据实际场景配置权重：

- **准时交付最大化：** 最小化拖期/延期交付工单数
- **设备利用率最大化：** 最大化瓶颈设备使用效率，减少空闲等待
- **换线/切换成本最小化：** 聚合同类产品减少换模具/换线次数
- **在制品库存最小化：** 减少WIP积压，均衡产线负载
- **最小完工时间（Makespan）：** 最小化最后一道工序的完成时间

### 2.3 约束条件

系统需支持的核心约束条件：

- **设备产能约束：** 每个设备在同一时间只能加工一个任务，需考虑设备可用时长和班次
- **工序顺序约束：** 同一工单下的工序须按工艺路线规定的先后顺序执行
- **物料齐套约束：** 任务开工前所需原材料/半成品必须齐备
- **人员技能约束：** 特殊工序需指定具备相应技能的操作人员
- **交货期约束：** 工单须在客户要求交期内完成
- **优先级约束：** 紧急插单/重点客户工单优先排产

### 2.4 基础数据需求

排程运行需要以下三类基础数据：

**工单数据（生产任务）**
- 工单号、产品编码、生产数量、计划开工日/交货日、优先级、是否可拆分

**工艺数据（产品BOM与路线）**
- 每道工序的标准加工时间、适用设备/工装、前后序关系、所需工种/技能等级

**资源数据（设备与人员）**
- 设备清单：设备编码、工作日历/班次、计划维护时段、加工能力限制
- 人员/班组：人员技能矩阵、出勤日历、可操作设备范围

### 2.5 输出物

排程智能体应输出以下交付物：

- **生产排程计划表：** 每个工单每道工序的开始/结束时间、分配设备、指派人员
- **甘特图：** 以可视化方式展示工序在设备时间轴上的排布
- **资源负荷图：** 各设备/人员的利用率曲线，识别瓶颈资源
- **异常预警报告：** 物料短缺、交期风险、设备超负荷等异常及时预警
- **KPI对比报告：** 排程方案 vs 人工排程的关键指标对比

## 3. 功能需求

### 3.1 功能架构总览

原型阶段聚焦四大核心功能模块：

#### 3.1.1 数据导入模块 (M1)

通过Excel模板导入工单、工艺、设备三类基础数据。

- 支持.xlsx格式，提供标准模板下载
- 自动校验数据完整性：必填字段检查、日期格式校验、引用完整性校验
- 异常数据行高亮提示并返回错误说明
- 小规模演示数据：10-20工单、3-5设备、3-5工序

#### 3.1.2 排程计算模块 (M2)

核心排程引擎，支持约束建模与多目标优化求解。

- **约束建模：** 将设备产能、工序顺序、交货期等业务约束转化为数学约束
- **目标函数：** 支持多目标加权组合（准时交付权重、设备利用率权重等）
- **求解器调用：** 集成OR-Tools CP-SAT求解器，支持限时求解（默认300s）
- **排程策略：** 先到先得（FCFS）、最早交期优先（EDD）、优化求解（Optimal）三种模式
- **结果输出：** 每道工序的起止时间、分配设备、产出结构化排程计划

#### 3.1.3 可视化展示模块 (M3)

以甘特图和仪表盘形式展示排程结果。

- **设备甘特图：** 横轴为时间，纵轴为设备，每道工序显示为色块并标注工单号
- **交期达成率统计：** 按时完成/延迟/提前完成的工单比例
- **设备利用率仪表盘：** 各设备负荷率百分比及瓶颈标识
- **交互操作：** 支持缩放时间轴、筛选工单/设备、悬停查看详情

#### 3.1.4 人机协同调整模块 (M4)

支持计划员对排程结果进行人工微调与What-if分析。

- **拖拽调整：** 甘特图上拖拽工序调整时间和设备
- **人工锁定：** 锁定特定工单/工序，重新排程时保持不变
- **What-if分析：** 模拟插单、设备故障等场景，评估对整体排程的影响
- **方案对比：** 并行展示"原方案"与"调整后方案"的KPI差异

## 4. 技术方案

### 4.1 技术选型路线对比

原型阶段提供三条技术选型路线，适用于不同团队条件和需求场景：

| 路线 | 方案说明 | 推荐场景 | 工期 |
|------|----------|----------|------|
| **A** | 低代码/商业APS平台（如JVS-APS、腾讯云MES排程模块等），通过配置而非编码实现排程，内置APS算法，可视化页面搭建，支持ERP/MES对接 | 小微企业，无专职开发团队，追求快速上线 | 1-3天配置 |
| **B** | Python + OR-Tools + Streamlit。以OR-Tools CP-SAT求解器为排程引擎，Streamlit搭建Web交互界面，完全自主可控 | 有技术团队，需要自主可控和可扩展性 | 1周原型 |
| **C** | 大模型（GPT/通义千问等）+ 低代码工具联动。大模型理解自然语言调度需求生成排程方案，对话式交互追问优化 | 探索型项目，希望验证AI+排程可行性 | 3-5天 |

### 4.2 推荐路线：路线B Python + OR-Tools

综合考虑可控性、可扩展性和实用性，推荐采用路线B作为主路线，理由如下：

- **开源可控：** Python生态成熟，OR-Tools为Google开源求解器，无商业授权风险
- **扩展性强：** 可逐步加入自定义约束、启发式算法、强化学习等高级能力
- **开发效率高：** Streamlit提供开箱即用的Web组件，数行代码即可构建交互界面
- **社区活跃：** OR-Tools文档完善，CP-SAT在排程/调度领域有大量参考案例

### 4.3 技术栈详细说明（路线B）

核心依赖：

| 组件 | 版本 | 用途 |
|------|------|------|
| Python | 3.10+ | 主开发语言 |
| OR-Tools | 9.x | CP-SAT约束规划求解器，负责排程建模与求解 |
| Streamlit | latest | Web前端框架，构建排程交互界面与可视化 |
| Pandas | latest | 数据处理与Excel读写 |
| Plotly | latest | 甘特图、仪表盘等交互式图表 |
| Pydantic | latest | 数据模型校验与配置管理 |

### 4.4 系统架构（四层架构）

系统采用四层分层架构，各层职责清晰、接口明确：

| 架构层次 | 功能职责 | 关键组件 |
|----------|----------|----------|
| **数据层** | 采集工单/工艺/设备/物料数据，完成数据清洗与结构化准备 | Excel导入模块、Pandas ETL管道、数据校验引擎 |
| **算法层** | 排程建模引擎+大模型辅助推理。将业务约束转化为数学模型，调用求解器计算最优排程方案 | OR-Tools CP-SAT、约束建模器、目标函数配置、大模型NLP接口（路线C） |
| **交互层** | 甘特图可视化、交互式仪表盘、人工调整界面 | Streamlit Web UI、Plotly图表、拖拽交互组件 |
| **执行层** | 排程结果下发至MES/ERP执行，实时采集设备状态反馈 | API网关、MES/ERP/WMS集成接口、Webhook通知 |

### 4.5 排程算法方案

根据问题规模和实时性需求，分层选用不同算法：

| 算法类型 | 原理 | 适用场景 | 工具 |
|----------|------|----------|------|
| **约束规划 (CP)** | 将排程问题建模为约束满足问题，利用约束传播+搜索求解 | 中小规模（<100工单），多约束离散排程 | OR-Tools CP-SAT |
| **启发式/元启发式** | 遗传算法 (GA)、模拟退火 (SA) 等进化算法逼近近似最优解 | 大规模问题（>500工单），精度要求可放宽 | optuna / 自研 |
| **强化学习 (RL)** | 训练智能体在动态环境中学习排程策略，适应频繁插单/异常场景 | 动态排程、高频插单、异常自适应 | Custom RL Env + PPO |

## 8. 成功标准与验收

### 8.1 原型阶段验收标准

1. 完成数据导入、排程计算、甘特图可视化三大核心功能的端到端跑通
2. 在20工单/5设备/5工序测试集上，排程求解时间≤5分钟
3. 排程方案在设备利用率和交期达成率两个核心KPI上不劣于人工排程
4. 计划员能独立完成"导入Excel → 点击排程 → 查看甘特图 → 导出结果"的完整操作流程

### 8.2 后续阶段考核指标

- **排程效率：** 人工排程耗时（4-6小时）→ 系统排程（<5分钟），效率提升>95%
- **设备利用率：** 提升5-10个百分点（基于实际产线历史数据对比）
- **交期达成率：** 提升至95%以上
- **计划员满意度：** 调研评分≥4.0分（5分制）

## 9. 附录

### 9.1 关键原则

以下原则贯穿整个搭建过程中：

- **不要一开始追求全覆盖/全自动**，先单点试点验证再推广
- **数据质量是关键** —— 统一编码、校验规则、时序准确性决定排程质量上限
- **人机协同是正确姿势** —— 系统生成排程方案，计划员做最终决策与异常处理
- **从最小目标开始** —— 先验证排程效率提升，逐步优化设备利用率和换线成本
